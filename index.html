CH.01

What Cloud Computing Actually Is

Cloud computing is the delivery of computing resources — servers, storage, databases, networking, software — over the internet, on demand, billed for what you use. The five traits that separate a "cloud" from a rented server:

01

On-demand self-service

Provision compute without calling a human.

02

Broad network access

Available over standard internet protocols, from anything.

03

Resource pooling

Multi-tenant hardware, dynamically assigned.

04

Rapid elasticity

Scale out in seconds, scale in just as fast.

05

Measured service

Metered like electricity — pay for consumption.

# The entire mental model, demonstrated:
# "give me a server" becomes an API call instead of a purchase order

curl -X POST https://api.cloudprovider.example/v1/instances \
  -H "Authorization: Bearer $API_TOKEN" \
  -d '{"type": "small", "image": "ubuntu-24.04", "region": "ap-south-1"}'

# response arrives in ~2 seconds with a running machine's IP
Why it matters: before cloud computing, "we need more capacity" meant buying physical servers with weeks of lead time. Now it means changing a number.
CH.02

Deployment Models

Deployment model answers: who owns the underlying infrastructure, and who else shares it?

Public Cloud

Owned by a third party (AWS, Azure, GCP), shared across customers. Lowest cost, fastest to start, least control.

Private Cloud

Dedicated to one organization, on-prem or hosted. Full control, higher cost, used for strict compliance needs.

Hybrid Cloud

Public and private linked together — e.g. keep customer data on-prem, burst compute to the public cloud.

Community Cloud

Shared by several organizations with common concerns — e.g. government agencies sharing a compliance-certified cloud.

CH.04

Virtualization & Hypervisors

Virtualization is the trick that makes clouds economical: one physical machine, sliced into many isolated virtual machines by a hypervisor.

  • Type 1 (bare-metal) — runs directly on hardware: VMware ESXi, Xen, Microsoft Hyper-V, KVM. Used inside real data centers.
  • Type 2 (hosted) — runs inside a host OS: VirtualBox, VMware Workstation. Used for local dev/testing.
# Creating a KVM virtual machine from the command line
virt-install \
  --name web01 \
  --memory 2048 --vcpus 2 \
  --disk size=20 \
  --os-variant ubuntu24.04 \
  --network network=default \
  --cdrom /isos/ubuntu-24.04.iso
Each VM believes it owns the whole machine — its own kernel, its own virtual CPU, disk, and NIC — while the hypervisor quietly schedules real hardware underneath.
CH.05

Containers & Docker

Containers skip the "full extra OS" cost of a VM. They share the host kernel and isolate only the process, filesystem, and network — starting in milliseconds instead of minutes.

# Dockerfile — packages a Node.js API into a portable image
FROM node:20-slim

WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev

COPY . .
EXPOSE 3000
USER node

CMD ["node", "server.js"]

# Build once, run anywhere the same way
docker build -t my-api:1.0 .
docker run -d -p 3000:3000 --name api my-api:1.0

docker ps
docker logs -f api
CH.06

Orchestration — Kubernetes

Once you have more than a handful of containers, you need something to schedule, heal, and scale them. Kubernetes is the industry-standard orchestrator.

# deployment.yaml — 3 replicas of the API, self-healing
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-api
spec:
  replicas: 3
  selector:
    matchLabels: {app: my-api}
  template:
    metadata:
      labels: {app: my-api}
    spec:
      containers:
        - name: my-api
          image: my-api:1.0
          ports: [{containerPort: 3000}]
          resources:
            requests: {cpu: "100m", memory: "128Mi"}
            limits:   {cpu: "500m", memory: "256Mi"}
---
apiVersion: v1
kind: Service
metadata:
  name: my-api-svc
spec:
  selector: {app: my-api}
  ports: [{port: 80, targetPort: 3000}]
  type: LoadBalancer

kubectl apply -f deployment.yaml
kubectl get pods -w
kubectl scale deployment my-api --replicas=6
CH.07

Cloud Storage & Databases

Object storage

Flat namespace of immutable blobs. S3, Azure Blob, GCS. Great for media, backups, logs.

Block storage

Raw disk volumes attached to a VM. EBS, Azure Disks. Needed for databases and OS disks.

File storage

Shared filesystem over NFS/SMB. EFS, Azure Files. Good for multi-VM shared access.

Managed databases

RDS, Cloud SQL, DynamoDB — provider handles backups, patching, failover.

CH.08

Cloud Networking

A VPC (Virtual Private Cloud) is your own isolated network slice inside the provider's data center — subnets, route tables, gateways, firewalls, all software-defined.

# A minimal VPC with a public and private subnet (Terraform)
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

resource "aws_subnet" "public" {
  vpc_id            = aws_vpc.main.id
  cidr_block         = "10.0.1.0/24"
  map_public_ip_on_launch = true
}

resource "aws_subnet" "private" {
  vpc_id     = aws_vpc.main.id
  cidr_block = "10.0.2.0/24"
}
  • Load balancer — spreads traffic across healthy instances (e.g. ALB, NGINX, HAProxy).
  • CDN — caches content at edge locations near users to cut latency (CloudFront, Cloudflare, Akamai).
  • DNS — routes domain names to the right load balancer or IP (Route 53, Cloud DNS).
CH.09

Serverless Computing

Serverless doesn't mean "no servers" — it means you never think about them. You ship a function; the platform runs it on-demand and bills per invocation.

# AWS Lambda handler — resizes an image whenever one lands in S3
import boto3
from PIL import Image
import io

s3 = boto3.client("s3")

def handler(event, context):
    record = event["Records"][0]["s3"]
    bucket, key = record["bucket"]["name"], record["object"]["key"]

    obj = s3.get_object(Bucket=bucket, Key=key)
    img = Image.open(io.BytesIO(obj["Body"].read()))
    img.thumbnail((200, 200))

    buf = io.BytesIO()
    img.save(buf, "JPEG")
    s3.put_object(Bucket=bucket, Key=f"thumbs/{key}", Body=buf.getvalue())

    return {"statusCode": 200}
You pay per millisecond of execution, not per idle hour. The tradeoff: cold starts, execution time limits, and less control over the runtime.
CH.10

Security & Identity (IAM)

Cloud security runs on the shared responsibility model: the provider secures the cloud (physical data centers, hypervisor); you secure what's in the cloud (data, access, configuration).

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::my-data-bucket/reports/*",
      "Condition": {
        "IpAddress": {"aws:SourceIp": "203.0.113.0/24"}
      }
    }
  ]
}
  • Least privilege — grant only the permissions a role strictly needs.
  • Encryption at rest & in transit — TLS on the wire, KMS-managed keys on disk.
  • MFA everywhere — especially on the account root/owner identity.
CH.11

DevOps, CI/CD & Infrastructure as Code

Cloud maturity shows up as automation: infrastructure defined in version-controlled code, deployed through a pipeline instead of a person clicking a console.

# .github/workflows/deploy.yml
name: deploy
on: {push: {branches: [main]}}

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t my-api:${{ github.sha }} .
      - run: docker push my-api:${{ github.sha }}
      - run: kubectl set image deployment/my-api my-api=my-api:${{ github.sha }}
"Infrastructure as Code" (Terraform, Pulumi, CloudFormation) means your VPC, databases, and clusters are text files reviewed in pull requests — reproducible and diffable, not a snowflake console someone clicked once in 2022.
CH.12

Cost Optimization & Monitoring

  • Right-sizing — matching instance size to actual CPU/memory usage instead of guessing high.
  • Reserved / spot capacity — commit ahead for a discount, or use spare capacity for interruptible workloads at up to 90% off.
  • Autoscaling — scale out under load, scale to zero when idle.
  • Observability — metrics, logs, and traces (CloudWatch, Prometheus + Grafana, Datadog) so you know why something costs what it costs or broke.
# Kubernetes Horizontal Pod Autoscaler
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: {name: my-api-hpa}
spec:
  scaleTargetRef: {apiVersion: apps/v1, kind: Deployment, name: my-api}
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target: {type: Utilization, averageUtilization: 60}
